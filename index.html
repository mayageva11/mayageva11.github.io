The problem it solves

The pain

A flaky test passes and fails on the same code - and every red build it causes trains your team to click "re-run" instead of investigating. Existing tools count pass/fail rates and tell you that a test is flaky; paid services like BuildPulse and Trunk.io charge for exactly that.

The FlakeHound difference

FlakeHound is free, open-source, and goes further: it normalises stack traces and clusters failures by their underlying cause - turning "23 failures over 3 weeks" into "4 unique bugs". It knows that a test failing 100% of the time since one commit isn't flaky, it's broken - and it gates CI on new regressions only, so the build fails once when a bug lands, not on every run until someone fixes it.

How it works

🔍

Root-cause clustering

Jaccard similarity + greedy clustering groups structurally identical failures. "23 failures = 4 bugs" - one cluster, one bug, even when it spans many tests and weeks of history.

🤖

AI hypothesis layer

Claude API or local Ollama (privacy-first, zero cost) annotates each cluster with a one-line root-cause hypothesis. Swappable provider interface - the deterministic core works identically without it.

🚧

Quarantine subcommand

Auto-isolates high-confidence flaky tests via surgical AST edits (ts-morph). Files one GitHub issue per test. Auto-releases after N consecutive clean runs - no manual cleanup needed.

📊

Self-contained dashboard

One HTML file with the full report embedded - open from disk, attach as a CI artifact, or publish directly to GitHub Pages. No server, no account, no dependencies.

The AI layer, in action

Optional and at the edge. A local Ollama model runs first (private, zero cost), the Claude API is the fallback — and since v0.5.0 a provider that can't start just degrades to "no hypotheses". The deterministic core never changes.

Inside the codebase

A living demo - the Flaky Shop 🛒

Not a screenshot - a running system. The Flaky Shop is a deliberately buggy e-commerce site with six planted defects: a random 0-35s render delay, a race condition in the cart, a hard regression in the receipt, simulated 502s, and more. A Playwright suite exposes them, and every 6 hours an unattended pipeline runs the tests against the live shop, analyses the accumulated history, applies quarantine verdicts through real pull requests, and republishes the dashboard. In my products

FlakeHound ships a pluggable AI hypothesis layer - a local Ollama model first (private, zero cost), the Claude API as fallback. QuestForge generates story scenes with Groq + Llama 3.1. NexusQA integrates the Claude API into its test platform.

⚙️

In my workflow

At work: Cursor for code generation and MCP-Git for automated PR creation, accelerating day-to-day automation. This site itself was built and deployed with Claude Code - AI-assisted, human-reviewed.

🧭

My philosophy

AI interprets - it is never the source of truth. I build deterministic cores with AI at the edges, and treat model failures as first-class cases: every AI path degrades gracefully, nothing hangs or lies.

💻

How I use Claude Code

This site was planned, built, tested, and deployed with Claude Code: the agent explores my repos, drafts a plan I approve, edits the code, verifies it with Playwright screenshots at phone, tablet, and desktop widths, and ships via git - every change human-reviewed.

🔁

Loop engineering

I run AI in tight feedback loops: generate → run → observe → refine. The model's output is never trusted blind - tests, screenshots, and CI close the loop, so every iteration is verified against reality before it lands.

✍️

How I write prompts

Prompts as specifications: full context up front, exact names and constraints, explicit do-nots, and labeled placeholders instead of invented facts. I write prompts the way I write test plans - unambiguous in, verifiable out.
